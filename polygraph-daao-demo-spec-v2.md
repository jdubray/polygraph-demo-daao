# Polygraph Demo Spec v2 — Dual-Authorization Action Order (DAAO)

**Status:** state machine + invariants **validated by running two independent implementations**
(Python 3 stdlib, Rust 1.95 std-only) generated by DeepSeek. Safety invariants confirmed to hold
on targeted probes; one real divergence found. This version supersedes v1 and is written as a
**build handoff** — the last section is the backlog for Claude Code.

**Audience:** the reviewer — built MySQL (durability, replication, exactly-once), ran HackerOne
(adversarial, combinatorial defects), now in EU defense (two-person integrity, assurance,
auditability). Every beat is chosen to hit one of those three.

**One-line framing:** *"This is the durability guarantee you shipped at MySQL, applied to the
combinatorial-bug problem you chased at HackerOne, wrapped in the two-person assurance case your
organization has to sign — on one eight-state machine, checked on every change."*

---

## 1. Scenario

An **Action Order** is proposed by one authority and may only be **executed** after **two
distinct other authorities** approve it, within an authorized time window. Execution triggers a
real-world effect exactly once. A recall must be possible right up until execution. Deliberately
**generic and unclassified** — `execute(order)` stays abstract so the reviewer maps it onto his own
high-consequence action.

**Roles:** Proposer (creates), Approver (two *distinct*, neither the proposer), Operator
(executes).
**Order data:** `id`, `proposer`, `approvers` (set of distinct IDs), `window = [start, end]`,
`state`.
**External effects (exactly-once, crash-safe):** `transmit(order)`, `execute(order)`.

---

## 2. State machine (validated)

```
        submit                approve×2 (distinct, in-window)         release
 DRAFT ────────▶ PENDING_APPROVAL ─────────────────────────▶ APPROVED ─────────▶ RELEASED
                    │      │                                    │                   │
                    │      │ expire                             │ (transmit once)   │ acknowledge
                    │      ▼                                    │                   ▼
                    │   EXPIRED ◀───────────────────────────────┴──────────── ACKNOWLEDGED
                    │                                                               │
                    │  abort (any pre-execution state ≥ PENDING_APPROVAL)           │ execute
                    ▼                                                               │ (execute once,
                 ABORTED ◀───────────────────────────────────────────────┐        ▼  in-window)
                                                                          │     EXECUTED
                                                                    abort │        │ close
                                                                          └──────  CLOSED
```

**States (8 + 2 terminal):** `DRAFT`, `PENDING_APPROVAL`, `APPROVED`, `RELEASED`, `ACKNOWLEDGED`,
`EXECUTED`, `CLOSED`, `ABORTED`, `EXPIRED`.

**Transition table**

| From | Event | Guard | To | Effect |
|---|---|---|---|---|
| DRAFT | `submit(p, window)` | — | PENDING_APPROVAL | set `proposer`, `window` |
| PENDING_APPROVAL | `approve(id)` | `id≠proposer ∧ id∉approvers ∧ now∈window`; `<2` distinct so far | PENDING_APPROVAL | `approvers ∪= {id}` |
| PENDING_APPROVAL | `approve(id)` | same guard; reaches 2 distinct | **APPROVED** | `approvers ∪= {id}` |
| PENDING_APPROVAL / APPROVED / RELEASED / ACKNOWLEDGED | `expire` | `now > window.end` | EXPIRED | — |
| APPROVED | `release` | `now ∈ window` | RELEASED | `transmit(order)` **once** |
| RELEASED | `acknowledge` | — | ACKNOWLEDGED | — |
| ACKNOWLEDGED | `execute` | `now ∈ window ∧ constraints()` | EXECUTED | `execute(order)` **once** |
| EXECUTED | `close` | — | CLOSED | — |
| PENDING_APPROVAL / APPROVED / RELEASED / ACKNOWLEDGED | `abort` | — | ABORTED | — |

## 3. Invariants — what the human owns

**Safety (must-never):**
- **S1 — Two-person integrity.** `authorized(order) ⇒ |distinct(approvers)| ≥ 2 ∧ proposer ∉
  approvers`, where `authorized` = state in {APPROVED, RELEASED, ACKNOWLEDGED, EXECUTED, CLOSED}.
- **S2 — Authorized window.** `execute` (and `release`) fire only when `now ∈ window`.
- **S3 — Exactly-once execution.** `execute(order)` emitted ≤ once across runs, retries, crashes;
  `EXECUTED` is not re-enterable.
- **S4 — No lost acknowledged order (durability).** Once `ACKNOWLEDGED`, the order reaches
  `EXECUTED`/`ABORTED`/`EXPIRED`; state survives a crash.
- **S5 — No forged authorization.** No transition re-enters an authorized state from one that
  reset/lost `approvers`. (By construction there are no reset/re-submit paths — confirm this stays
  true as the machine grows.)

**Liveness:**
- **L1 — Recall reachable.** From every state ≥ `PENDING_APPROVAL` and pre-`EXECUTED`, an `abort`
  reaches `ABORTED`. (Note: `DRAFT` cannot abort — acceptable, it is pre-submission.)
- **L2 — Progress.** Every order reaches a terminal state.

**Mapping for the reviewer:** S3 + S4 = exactly-once-under-failure (MySQL). S1 + S2 = the two-person
rule + ROE envelope (assurance case). S5 = manufacturing authorization by looping the graph
(red-team).

## 4. What's already built (in `daao/`)

Confirmed by running both builds head-to-head with a conformance harness and an S1 gate:

| Probe | Input | Result (both languages) | Verdict |
|---|---|---|---|
| Self-approval | proposer also approves | stalls `PENDING_APPROVAL` | **S1 ✓** |
| Duplicate approver | same approver twice (redelivery) | stalls `PENDING_APPROVAL` | **S1 ✓** |
| Out-of-window execute | execute after `window.end` | stalls `ACKNOWLEDGED`, no effect | **S2 ✓** |
| Redelivered release/execute | release ×2, execute ×3 | one `transmit`, one `execute` | **S3 ✓** |
| Crash / recover | kill mid-trace, resume | resumes, effects once | **S4 ✓** |

**Both languages were semantically identical on every trace.** DeepSeek got the safety logic
right — deduped approvers, gated the window on the event's own timestamp. The naive "model writes
the two-person bug" beat did **not** happen; do not pretend it did.

Files: `safety_engine.py`, `safety_engine.rs` (+ compiled `safety_engine_rs`),
`safety_engine_buggy.rs` (+ `safety_engine_buggy`), `harness.py`, `check_s1.py`, and traces
`happy_path`, `p1_self_approval`, `p2_dup_approver`, `p3_out_of_window`, `p4_redeliver_execute`,
`cr` (crash), `multi` (determinism), `ce_dup` (planted counterexample).

## 5. The demo — two acts (revised to the real result)

### Act 1 — Verification vs. vibes (authentic, un-planted) — *his MySQL + audit nerve*
DeepSeek's README asserts the two builds produce identical output and *"any divergence is a
defect."* It's wrong, and the model didn't notice.

*Do:* byte-diff the two outputs on `multi.json`, and run the Rust build three times.
*What appears:*
```
Python  run 1/2: ['zeta','alpha','mike','bravo']   (stable, insertion order)
Rust    run 1:   ['zeta','mike','alpha','bravo']
Rust    run 2:   ['mike','bravo','zeta','alpha']    (different every run)
Rust    run 3:   ['mike','alpha','bravo','zeta']
```
The Rust build stores orders in a `HashMap`; its output **and its persisted `store.json`** are
non-deterministic. *Watch this:* a naive diff flags this as a failure — and as *flaky* — a false
positive on a difference that carries no behavioral meaning. The contract-level check adjudicates
correctly: equivalent on `(state per order, ordered effect log)`, divergence isolated to map
serialization, flagged **benign for behavior**. **Callout:** but the same non-determinism is a
**real** defect the moment `store.json` is an audit record you hash, sign, or replicate by
checksum — MySQL replication checksums, a defense audit log that must verify deterministically. A
diff can't tell you which difference matters; the gate can. *That is the difference between a green
checkmark and a proof.*

### Act 2 — Language-agnostic catch (planted, and say so) — *his HackerOne nerve*
*Do:* run the 4-step counterexample `ce_dup.json` through the clean builds and the planted Rust.
*What appears:*
```
[CLEAN-PY ]  x1: PENDING_APPROVAL | effects: []            ✓ S1 holds
[CLEAN-RS ]  x1: PENDING_APPROVAL | effects: []            ✓ S1 holds
[PLANTED-RS] x1: CLOSED | effects:[transmit, execute]      ✗ S1 VIOLATED
             proposer=Alice, approvers=['Bob','Bob'] -> distinct=1; execute() emitted
```
*Watch this:* same contract, same gate — it passes both clean builds and catches the buggy one,
with the reproduction. **Callout — disclose it:** "The model got this right this round, so I
reintroduced the classic mistake to show the catch. The point isn't that *this* model failed —
it's that the bug class is real, models do write it, and the gate is the net regardless of
language."

### Act 3 — polyvers: evolve the live fleet — *the wedge*
*Do:* take the **clean** build, tighten the rule **two approvers → three**, and re-check against
orders already in flight at `APPROVED` (approved under the 2-of-N rule). *Watch this:* the gate
seeds the exhaustive check with the *existing* in-flight states and returns the shortest path by
which an already-`APPROVED` order would `execute` under the new 3-approver invariant — blocking the
deploy and proposing the migration (re-open affected orders to `PENDING_APPROVAL`). **Callout:** in
defense, changing the rule means re-accrediting the whole system; here the deploy carries the
proof. *(This beat is not yet built — see §7.)*

## 6. Honesty notes (keep these in the room)
- `check_s1.py` is a hand-written **stand-in** for the Polygraph gate: it reads engine output and
  asserts the S1 predicate on crafted traces. It demonstrates the catch, but it is **not**
  exhaustive model-checking. In the real demo the gate is the **`polygraph` Claude Code plugin**
  run on this codebase, so the counterexample comes from state exploration, not one trace.
- The probes are **targeted, not exhaustive** — they confirm the invariants hold *on those paths*.
  "Holds on my probes" is testing; "holds on every reachable state" is what the gate claims and
  what you sell. State the gap plainly; it's your whole value.

## 7. Handoff backlog for Claude Code

Polygraph ships as **Claude Code plugins** — `polygen`, `polynv`, `polyrun`, `polygraph`,
`polyvers`. The backlog below is a sequence of plugin invocations on the `daao/` codebase, each
step making the demo more real. Map each engine to its beat rather than hand-rolling checks.

1. **`polynv` — lock the invariants.** Elicit/confirm S1–S5, L1–L2 as the DAAO contract by judging
   the scenarios in §4, and mutation-grade them so each rule has teeth. Output is the gate the
   other engines run against — replacing the `check_s1.py` stand-in.
2. **`polygraph` — audit the builds.** Run the `polygraph` plugin over `safety_engine.py`,
   `safety_engine.rs`, and the "legacy" `safety_engine_buggy.rs`: generate the independent spec,
   replay traces, model-check. Act 2's catch and the counterexample should be **plugin-produced**
   from state exploration, not the crafted `ce_dup.json`. Confirm it clears the two clean builds and
   flags the planted one.
3. **`polyvers` — build Act 3 (the wedge).** Version the contract (2→3 approvers), seed the check
   with in-flight states from a live `store.json`, produce the shortest migration-violating path,
   and gate the deploy. Currently the only unbuilt act.
4. **`polygen` — regenerate one engine to spec (optional).** Have `polygen` author a fresh DAAO
   implementation that is checkable from the first line and self-repairs to the §3 checks — a clean
   "verified-by-construction" counterpoint to the two DeepSeek builds.
5. **`polyrun` — make durability explicit (optional).** Move effect emission to transition-time I/O
   (not deferred-to-print); exactly-once now requires commit-then-emit ordering, and a faithful spec
   of "effect = external I/O" exposes the emit/persist seam. Strong advanced beat for a database
   audience.
6. **Decide the serialization finding's fate.** Either (a) canonicalize output in both engines
   (sort keys) and keep the non-determinism as an Act-1 *narrated* finding on the pre-fix build, or
   (b) leave it and let the `polygraph` audit in step 2 surface it. Recommend (a): fix it, but demo
   the before.
7. **One-command runner.** A single script (or Claude Code session recipe) that runs Act 1
   (determinism diff), Act 2 (`polygraph` clean-vs-planted catch), and Act 3 (`polyvers` gate)
   end-to-end, so the live demo is deterministic and repeatable.
